### Q1. HTML이 렌더링 중에 JavaScript가 실행되면 멈추는 이유는 무엇일까요?

> **importance_score**: 4/5 | **빈출**: high | **출처**: [Frontend-Interview-Questions](https://github.com/Esoolgnah/Frontend-Interview-Questions)

**모범 답변:**
JS 엔진과 렌더링 엔진은 단일 스레드를 공유합니다. JS가 DOM·CSSOM을 수정할 수 있기 때문에, 파서가 `<script>` 태그를 만나면 파싱을 중단하고 JS를 먼저 다운로드·실행한 뒤 다시 파싱을 재개합니다. 이를 방지하려면 `async`/`defer` 속성을 활용하거나 `<script>` 태그를 `</body>` 직전에 위치시킵니다.

### Q2. CSS `<link>` 태그를 `<head>` 안에, JS `<script>` 태그를 `</body>` 직전에 위치시키는 이유는 무엇인가요?

> **importance_score**: 4/5 | **빈출**: high | **출처**: [frontend-tech-interview](https://github.com/saseungmin/frontend-tech-interview)

**모범 답변:**
CSS는 렌더링 차단(render-blocking) 리소스이므로, 최대한 빨리 CSSOM을 완성해야 첫 화면을 그릴 수 있어 `<head>` 안에 위치시킵니다. JS는 DOM 파싱을 블로킹하므로, 페이지 콘텐츠가 먼저 로드된 후 실행되도록 `</body>` 직전에 두어 사용자가 흰 화면(White Screen)을 보는 시간을 최소화합니다.

### Q3. `<script>`, `<script async>`, `<script defer>` 사이의 차이점을 설명해주세요.

> **importance_score**: 4/5 | **빈출**: high | **출처**: [frontend-tech-interview](https://github.com/saseungmin/frontend-tech-interview)

**모범 답변:**
- **기본 `<script>`**: HTML 파싱 중단 → 스크립트 다운로드 및 즉시 실행 → 파싱 재개
- **`async`**: 다운로드는 파싱과 병렬 진행, 다운로드 완료 즉시 파싱 중단 후 실행 (실행 순서 보장 X)
- **`defer`**: 다운로드는 파싱과 병렬 진행, HTML 파싱이 완전히 완료된 후 선언 순서대로 실행

독립적인 외부 스크립트는 `async`, DOM 의존 스크립트는 `defer`가 적합합니다.

### Q4. 브라우저의 렌더링 과정에서 자바스크립트는 어떻게 동작하나요?

> **importance_score**: 4/5 | **빈출**: high | **출처**: [prepare_frontend_interview](https://github.com/junh0328/prepare_frontend_interview)

**모범 답변:**
HTML 파싱 중 `<script>` 태그를 만나면 렌더링 엔진이 JS 엔진에 제어권을 넘깁니다. JS가 DOM API를 통해 DOM·CSSOM을 변경하면 Render Tree 재생성 → Layout(Reflow) → Paint(Repaint) 과정이 다시 수행됩니다. `requestAnimationFrame`이나 `documentFragment`를 활용해 불필요한 리플로우를 줄이는 것이 성능 최적화의 핵심입니다.

### Q5. 브라우저의 렌더링 원리에 대해 설명해주세요.

> **importance_score**: 3/5 | **빈출**: high | **출처**: [Frontend-Interview-Questions](https://github.com/Esoolgnah/Frontend-Interview-Questions)

**모범 답변:**
브라우저 렌더링은 크게 5단계로 이루어집니다: ① HTML 파싱 → **DOM 트리** 생성, ② CSS 파싱 → **CSSOM 트리** 생성, ③ DOM + CSSOM 결합 → **Render Tree** 구성 (display:none 요소 제외), ④ 각 노드의 크기·위치 계산 → **Layout(Reflow)**, ⑤ 실제 픽셀로 그리기 → **Paint**, ⑥ GPU 레이어 합성 → **Composite**. JS 실행이나 스타일 변경 시 Layout 또는 Paint 단계부터 재수행됩니다.
